# Legacy 与当前 v2 缺口对比

## 结论口径

本文件对比两个目录：

- 老版本参考项目：`E:\Hachimi\WebHachimi`
- 当前重构项目：`E:\WebForAll\WebForAll`

当前任务是重构，不是原样迁移。老版本用于确认功能意图、交互细节和技术原型；当前 v2 应该承接有价值的能力，但不继承旧的单文件结构、拥挤 UI 和会混淆运行态/编辑态的行为。

本轮证据主要来自源码、文档、脚本和静态对比。子任务已确认没有发现 P0 级“当前版无法启动/构建”的缺口；当前主要问题是功能承接不完整、编辑器事务化不完整、以及部分旧版细节尚未转成新的产品形态。

## 重要修正

老版本会把部分非持久运行时对象也放进运行世界和场景树，例如“普通攻击判定”。这不是当前 v2 应该照搬的行为。

当前要求是：一个物体没有实际出现时，不应该被场景树查找出来，也不应该像普通场景实体一样被选择。也就是说，“普通攻击判定”这类对象应该保留为模板、调试对象或运行时瞬时对象，而不是常驻编辑实体。

需要补的是另一条能力：

- 模板/调试面板可以编辑攻击判定、投射物、特效等运行时模板。
- AI 可以通过明确的“模板目标”修改这些模板参数。
- 场景树仍然只展示实际存在的可编辑场景对象。

## 当前 v2 已经优于老版的部分

### 模块化架构

老版核心是 `index.html`、`styles.css`、`app.js` 组成的大型单页工具。当前 v2 已经拆成 TypeScript 模块，至少包括项目模型、运行时、AI、测试、Pixi 渲染器、编辑器 shell、面板视图、持久化控制器、任务工作流等。

这条方向应该继续保持，不能回到 `app.js` 式堆叠。

### 项目模型与审计链

当前 v2 的项目模型已经包含：

- scenes
- entities
- resources
- tasks
- transactions
- snapshots
- testRecords
- autonomyRuns

这比老版更适合后续做 AI 修改、diff、回滚、自主测试和发布链路。

### 运行时与测试闭环

当前 v2 已经有固定步进、运行态冻结、快照、战斗事件、瞬时实体、自主测试和 smoke 脚本。老版更多是可玩的编辑器原型，当前 v2 更接近“可被测试和审计的引擎”。

### 独立玩家端

当前 v2 已经有 `player.html` 和 `src/player/*`。这是正确方向：编辑器可以复杂，玩家端必须轻量，不依赖编辑器 UI、Node、npm 或 TypeScript 运行环境。

## P0 缺口：必须补齐

### 1. 编辑器 CRUD 和属性编辑不足

老版可以创建模板对象、删除对象、编辑属性、调整 body/collider/render/controller 等常见字段。当前 v2 的检查器更多是展示信息，创建/删除/属性编辑入口不完整。

需要补齐：

- 创建实体：玩家、平台、敌人、金币、文本、触发区等基础模板。
- 删除实体。
- 编辑 transform：位置、尺寸、旋转。
- 编辑 render：颜色、形状、文本、可见性。
- 编辑 body/collider：body mode、碰撞开关、trigger、layer/mask。
- 编辑 behavior：玩家、敌人、金币、危险物、攻击、格挡等行为参数。

所有这些操作都必须走 transaction，而不是直接改 runtime world。

### 2. Undo/Redo 能力没有完整暴露

当前项目层已经有 undo/redo 能力，但 v2 UI 没有把它作为一等入口暴露出来。老版顶部有明确的撤销/重做按钮。

需要补齐：

- 顶部或窗口菜单里的 undo/redo。
- 快捷键。
- 状态提示：当前事务名称、是否可撤销、是否可重做。
- 对属性编辑、创建、删除、资源绑定、任务编辑都生效。

### 3. 事务化覆盖面还不够

当前 transform、文件夹移动等已经开始走 transaction，但仍不够。下一阶段必须把所有编辑器写操作纳入项目事务层。

最低覆盖：

- 属性面板修改。
- 创建实体。
- 删除实体。
- 文件夹创建/重命名/删除。
- 资源创建、绑定、解绑。
- 任务创建、编辑、状态变更。
- 模板参数修改。

### 4. 资源系统没有完全承接老版语义

老版资源系统不仅是附件列表，还包含 intent、usage、timing、placement、tags、notes、attachments、resource-to-entity 等语义。当前 v2 有 resources 和 bindings，但还没有完全恢复老版的“资源作为 AI 上下文资产”的价值。

需要补齐：

- 资源导入：文件、拖拽、剪贴板。
- 资源预览：图片、材质、GIF/动图或至少帧预览。
- 资源语义：用途、意图、使用时机、放置规则、绑定槽位。
- 资源绑定到实体。
- 从资源生成实体。
- AI context 中输出资源和绑定关系。

### 5. 注释/标注系统缺失

老版有 scene/entity/resource annotations、canvas pin、focus、create/delete 等设计。当前 v2 只有超级画笔上下文里的临时注释概念，还没有形成持久标注系统。

需要补齐：

- 场景注释。
- 实体注释。
- 资源注释。
- 画布 pin。
- 注释与 AI 任务上下文关联。
- 注释面板和定位。

### 6. JSON 导入/导出和持久化边界需要产品化

老版有 JSON import/export 和 server/localStorage fallback。当前 v2 已经整理了 `/api/v2/project`，但 UI 入口和交付说明还不够完整。

需要补齐：

- 导出当前 v2 project JSON。
- 导入 v2 project JSON。
- 保存失败时的本地 fallback 提示。
- legacy `/api/project` 与 v2 `/api/v2/project` 的边界说明。
- 发布包与源码包边界。

## P1 缺口：应尽快优化

### 1. 场景树与选择细节

老版有更多画布微交互，例如多选、框选、聚焦、锁定、可见性、上下文菜单、对象列表联动。当前 v2 已经有基础树和拖动，但还不完整。

建议补齐：

- 多选与框选。
- 锁定/解锁。
- 显示/隐藏。
- 聚焦到对象。
- 右键菜单。
- 对象重命名。
- 文件夹管理。

### 2. 面板系统还需要继续打磨

当前已按要求做成独立浮动/停靠窗口，并且游戏模式下收起窗口，这是正确方向。仍需要补：

- 面板 resize。
- 面板 reset layout。
- dock 区域可视反馈。
- 同边堆叠和拆分时的细节。
- 小屏幕布局。
- 状态持久化。

不建议恢复老版左侧固定工具栏，因为用户明确要求用窗口按钮统一管理窗口来节省空间。可以保留非常少量的高频模式按钮，但默认不应该重新占用左侧空间。

### 3. 模板编辑入口

当前隐藏非持久模板是对的，但需要单独的模板编辑入口。否则 AI 和用户很难调整攻击判定、投射物、特效等运行时生成对象。

建议新增：

- 模板窗口。
- 运行时调试窗口。
- 攻击判定模板参数编辑。
- 模板预览生成。
- AI 修改模板时的 diff 预览。

### 4. AI 计划与上下文预览不足

老版有复制 AI context、生成计划、应用计划等显性入口。当前 v2 的任务执行链更强，但用户很难看清 AI 到底准备改什么。

需要补齐：

- 复制 AI context。
- 计划预览。
- diff 预览。
- dry-run 结果。
- 失败原因和自动后续任务。
- 回滚按钮。

### 5. 图像/资源渲染仍偏基础

当前 Pixi 渲染器主要画基础形状、颜色和标签。老版资源系统有把图片/材质/GIF 与实体关联的方向。

需要补齐：

- 资源图片作为实体外观。
- 附件 transform。
- 材质或贴图预览。
- 缺失资源占位。
- 玩家端资源加载路径。

### 6. 文案与编码需要巡检

部分文档和 UI 字符串在 PowerShell 下显示出乱码风险，部分源码注释/提示也需要确认是否为真实 mojibake 或只是终端编码问题。

需要做一次专项：

- 检查所有 `.md`、`.ts`、`.html`、`.css` 的编码。
- 统一为 UTF-8。
- 修复真实乱码字符串。
- 保证浏览器内中文正常显示。

## P2 缺口：后续增强

### 1. 物理系统增强

当前物理仍是基础 AABB。后续常见 2D 游戏需求需要：

- swept AABB。
- one-way platform。
- trigger/sensor 事件。
- collision layer/mask 完整测试。
- 地面/墙面检测。
- 高速物体防穿透。
- hitbox/hurtbox 生命周期。

### 2. 玩家端发布链路

当前玩家端已经独立，但还需要更完整的发布链：

- 项目打包。
- 资源打包。
- 只读 player runtime。
- 移动端触控体验。
- 载入失败/资源缺失提示。

### 3. 跨平台脚本

旧版和当前仍有 Windows 风格脚本痕迹。后续需要：

- 避免 `set ...&&` 这种 Windows-only 命令。
- 自动打开浏览器逻辑跨平台或默认关闭。
- README 区分 Windows/macOS/Linux。

### 4. 浏览器目标层验证脚本化

当前已经多次做过前台和 CDP 验证，但应该沉淀成可重复脚本：

- 打开 v2。
- 验证画布非空。
- 验证滚轮缩放。
- 验证面板拖动和 dock。
- 验证游戏模式隐藏窗口。
- 验证创建/编辑/撤销/保存/加载。

## 不建议恢复的旧版设计

以下旧版行为不应该原样迁回：

- 不恢复大型 `app.js` 单文件结构。
- 不把所有功能堆回固定左右栏。
- 不让未出现的非持久运行时模板混入普通场景树。
- 不把 AI 描述成通用大模型式自由改代码能力；当前仍应定位为规则式任务规划器 + 自动测试器。
- 不让玩家端依赖编辑器 UI 或开发环境。

## 下一阶段执行队列

建议按下面顺序继续：

1. 做可编辑检查器：先覆盖 transform、render、body、collider 的常用字段，全部走 transaction。
2. 加实体创建/删除：恢复老版模板能力，但用新 UI 和事务系统实现。
3. 暴露 undo/redo：按钮、快捷键、状态提示和 smoke 测试。
4. 建模板窗口：专门管理攻击判定、投射物、特效等非持久模板。
5. 扩展资源系统：导入、预览、绑定、资源生成实体、AI context。
6. 建持久注释系统：scene/entity/resource annotations 与 canvas pin。
7. 补 JSON import/export。
8. 把以上关键路径纳入浏览器回归脚本。

优先级判断：先让编辑器真正能“创建、编辑、撤销、保存、加载”，再去补更复杂的 AI 和资源体验。否则 AI 修改链会缺少稳定的人工编辑基础。
